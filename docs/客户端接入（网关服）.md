# 客户端接入（gatesvr）

本文档记录 gatesvr 的客户端接入设计。网络、编解码、分发和 session 能力位于 `internal/core`，gatesvr 按本文件约定完成接入装配。

## 组件职责

| 组件 | 当前位置 | 职责 |
|---|---|---|
| Acceptor | `internal/core/acceptor` | 接收 TCP 或 WebSocket 连接 |
| Connection | `internal/core/conn` | 异步读写 socket，维护最近收包时间 |
| Codec | `internal/core/codec` | Packet 和 Message 编解码 |
| SessionManager | `internal/core/session` | 连接态、认证态、UID 绑定、serverType 到 NodeID 的亲和绑定 |
| Dispatcher | `internal/core/dispatcher` | 按 CmdID 路由到本地 handler 或转发函数 |
| RPC Core | `internal/core/rpc` | gate 转发后端请求、接收回包并回调 |
| GateRemote | `protocol/remote/gate_remote.proto` | 后端业务服主动调用 gate 的 Remote 接口 |

典型链路：

```text
Acceptor -> Connection -> Codec -> GateDispatcher
                                      |
                        ┌─────────────┴─────────────┐
                        | 本地 handler                | 转发到后端
                        v                            v
                   Middleware                    rpc.Core -> RouterAgent
```

## Acceptor

`internal/core/acceptor` 抽象连接接入：

```go
type Acceptor interface {
    Listen() error
    Accept() <-chan conn.Connection
    Close() error
}
```

当前已有 TCP 和 WebSocket acceptor。accept goroutine 收到连接后，应把连接建立事件投递到 App 主循环，由 SessionManager 创建 Session。

## Connection

`internal/core/conn` 的连接对象负责异步读写：

```go
type Connection interface {
    Send(data []byte)
    Close() error
    RemoteAddr() string
    Done() <-chan struct{}
    LastRecvUnixNano() int64
    TouchRecv()
}
```

约定：

- read goroutine 只读网络和解码 Packet，不直接修改业务状态。
- 读到业务数据后通过 `app.Post` 投递到主循环。
- `Send` 写入连接内部写队列，由 write goroutine 实际写 socket。
- `lastRecv` 使用原子时间戳，供 heartbeat 检测使用。
- 逻辑清理一律回到主循环执行。

## Codec

外层 Packet 解决 TCP 粘包：

```text
| type(1B) | length(3B, big-endian) | body |
```

Packet 类型：

| 值 | 名称 | 方向 |
|---|---|---|
| `0x01` | Handshake | 客户端到服务端 |
| `0x02` | HandshakeAck | 服务端到客户端 |
| `0x03` | Heartbeat | 双向 |
| `0x04` | Data | 双向业务数据 |
| `0x05` | Kick | 服务端到客户端 |

Data body 内部是 Message：

```text
Request:  | type(1B) | SeqID(2B) | CmdID(4B) | body |
Response: | type(1B) | SeqID(2B) | CmdID(4B) | err_code(4B) | body |
Notify:   | type(1B) | CmdID(4B) | body |
```

`SeqID` 是客户端连接内的 `uint16` 请求序列。`err_code` 使用 `protocol/common/errcode.proto` 的 `ErrCode` 值。

## Session

当前 `internal/core/session.Session`：

```go
type Session struct {
    ID         string
    UID        int64
    ConnID     string
    Conn       conn.Connection
    Authed     bool
    BoundNodes map[uint32]uint32
}
```

SessionManager 维护两个索引：

- `byConn`：连接 ID 到 Session。
- `byUID`：UID 到 Session。

`BindSession` 会写入 UID、置 `Authed=true`，并合并亲和节点表。若同一 UID 已绑定旧 session，会清理旧 session 的 UID 和认证状态，避免 `byUID` 索引冲突。

`BoundNodes` 语义：

```text
serverType -> nodeID
```

gate 转发客户端请求时，如果命中 `BoundNodes[serverType]`，应使用 DIRECT 路由；未命中时按连接或业务 key 做 hash 选点。

## GateDispatcher

`internal/core/dispatcher.GateDispatcher` 处理客户端 Packet：

```go
switch pkt.Type {
case PacketHandshake:
    handshakeHandler(conn, pkt.Body)
case PacketHeartbeat:
    conn.TouchRecv()
case PacketData:
    msg := codec.DecodeMessage(pkt.Body)
    sess := sessions.GetByConnID(conn.RemoteAddr())
    dispatcher.Dispatch(sess, &msg)
}
```

`Dispatcher.Dispatch` 的逻辑：

1. 按 `msg.CmdID` 查路由表。
2. 如果路由目标 serverType 不是本服务，调用 `ForwardFunc` 转发。
3. 否则查本地 handler 并执行 middleware 链。
4. 未命中路由或 handler 时返回明确错误码。

当前 `ForwardFunc` 是扩展点，具体 gate 转发到 RouterAgent 的逻辑需要在 gatesvr 装配时注入。

## 路由表

客户端入口路由表由 `tools/gen_routes` 从 `protocol/handler/*.proto` 生成到 `protocol/gen/routes.go`：

```go
var RouteTable = map[uint32]RouteEntry{
    2050: {ServerType: serverTypeLobby, Route: "LobbyHandler/ClaimReward", RspCmdID: 2051},
    2052: {ServerType: serverTypeLobby, Route: "LobbyHandler/SyncPos", RspCmdID: 0},
}
```

`RspCmdID=0` 表示该上行是 notify，不需要客户端响应。

## 上行转发

设计链路：

```text
客户端 Data -> gate DecodeMessage -> RouteTable[CmdID]
    -> 本地 handler 或 ForwardFunc
    -> rpc.Core.Call / Send
    -> RouterAgent
    -> 目标业务服 Handler
    -> Reply 回包
    -> gate pending 查原连接
    -> conn.Send(Response)
```

关键边界：

- gate 转发时不解析业务 body，只看 CmdID 和路由表。
- `_Req` 需要在 gate 维护 pending，回包后写回原连接。
- `_Ntf` 直接 fire-and-forget，不进 pending。
- 所有 pending 操作应在主循环完成，超时和断线后的迟到回包按查不到 pending 丢弃。

pending 表和 forward-to-backend 逻辑属于 gatesvr 转发模块职责，应与 `ForwardFunc`、`rpc.Core` 和 RouterAgent 传输层一起装配。

## GateRemote

后端业务服主动调用 gate 的接口定义在 `protocol/remote/gate_remote.proto`：

```protobuf
service GateRemote {
  option (protocol.common.kind) = BACKEND;
  option (protocol.common.server_type) = ST_GATESVR;

  rpc SendToClient(RPC_SendToClient_Ntf) returns (google.protobuf.Empty);
  rpc BindSession(RPC_BindSession_Ntf) returns (google.protobuf.Empty);
  rpc SetBound(RPC_SetBound_Ntf) returns (google.protobuf.Empty);
}
```

GateRemote 的消息字段应承载下行业务所需的最小信封信息，gate 侧 Remote 实现负责：

- `SendToClient`：按 UID/session 找连接，发送指定 CmdID 和 payload。
- `BindSession`：登录成功后绑定 UID、置认证状态、写入亲和节点。
- `SetBound`：写入或更新某个 serverType 的目标 NodeID。

## 握手和心跳

当前握手由 `GateDispatcher.SetHandshakeHandler` 注入业务回调。基础约定：

```text
Client -> Server : Handshake
Server -> Client : HandshakeAck
```

Heartbeat 包由 gate 直接更新 `LastRecvUnixNano`，不进入业务 dispatcher。超时检测应由 gatesvr 模块用 timewheel 或定时任务在主循环内触发，最终调用 `SessionManager.OnTimeout` 和 `conn.Close`。
