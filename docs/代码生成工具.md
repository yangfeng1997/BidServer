# 代码生成工具

本文档记录 BidServer 的代码生成链路。协议层采用 `handler` / `remote` 分层，当前主服务为 `gatesvr`、`lobbysvr`、`routeragent`。

## 工具总览

| 工具 | 路径 | 职责 |
|---|---|---|
| `configgen` | `tools/configgen` | 从 `config/schema/*.proto` 生成 `config/gen` 配置结构、loader、entry、reload、validate；配置系统详见 `docs/配置系统.md` |
| `gen_routes` | `tools/gen_routes` | 从 `protocol/handler/*.proto` 生成客户端入口 CmdID 路由表 |
| `protoc-gen-svcstub` | `tools/protoc-gen-svcstub` | protoc 插件，生成 handler stub、remote stub 和 typed RPC stub |
| `servergen` | `tools/servergen` | 生成服务壳：`cmd/<svc>`、`internal/server/<svc>`、配置 schema 和 yaml |
| `gen_proto.sh` | `scripts/gen_proto.sh` | 编排协议 `.pb.go`、handler/remote/RPC stub、路由表生成 |
| `config.py` / `config_bake.py` / `build.py` | `scripts/` | 环境配置烘焙和构建编排 |

生成产物不要手工改业务语义。协议以 `.proto` 为事实源，配置以 `config/schema` 为事实源。

## 协议 Option

协议生成链路使用 `protocol/common/node.proto` 和 `protocol/common/options.proto`。

`ServerType` 当前定义：

```protobuf
enum ServerType {
  ST_UNKNOWN = 0;
  ST_GATESVR = 1;
  ST_LOBBYSVR = 2;
  ST_ROOMSVR = 3;
  ST_MATCHSVR = 4;
  ST_ONLINESVR = 5;
  ST_ROUTERAGENT = 6;
}
```

生成链路按实际输入的 handler / remote proto 生成路由、stub 和 RPC 入口。`ServerType` 枚举是协议基础类型，新增服务类型后应同步协议、生成器映射和服务实现。

`options.proto`：

```protobuf
enum ServiceKind {
  FRONTEND = 0;
  BACKEND = 1;
}

extend google.protobuf.ServiceOptions {
  ServiceKind kind = 50001;
  ServerType server_type = 50002;
}

extend google.protobuf.MessageOptions {
  uint32 cmd_id = 50003;
  bool no_auth = 50004;
}
```

语义：

- `FRONTEND`：客户端入口协议，service 名必须以 `Handler` 结尾。
- `BACKEND`：服务间 RPC 协议，service 名必须以 `Remote` 结尾。
- `cmd_id` 挂在 message 上，描述客户端帧 CmdID。
- `no_auth` 挂在客户端上行 message 上，进入鉴权白名单。

## gen_routes

`tools/gen_routes` 扫描 `protocol/handler/*.proto`，只处理 `kind = FRONTEND` 的 service。

输入：

```text
protocol/handler/*.proto
```

输出：

```text
protocol/gen/routes.go
```

生成内容：

```go
type RouteEntry struct {
    ServerType uint32
    Route string
    RspCmdID uint32
}

var RouteTable = map[uint32]RouteEntry{...}
var AuthWhitelist = map[uint32]bool{...}
```

生成逻辑：

- route = `ServiceName/MethodName`，例如 `LobbyHandler/ClaimReward`。
- `CmdID` 来自 input message 的 `cmd_id`。
- `RspCmdID` 来自 output message 的 `cmd_id`。
- `returns google.protobuf.Empty` 时 `RspCmdID=0`。
- input message 有 `no_auth=true` 时进入 `AuthWhitelist`。
- `cmd_id=0` 保留，生成器拒绝。
- `cmd_id` 全局重复时生成器拒绝。

当前 serverType 常量映射只为主链路生成：`ST_GATESVR`、`ST_LOBBYSVR`、`ST_ROUTERAGENT`。

## protoc-gen-svcstub

`tools/protoc-gen-svcstub` 是自研 protoc 插件。它读取 `protocol/handler/*.proto` 和 `protocol/remote/*.proto`，按 `protocol.common.kind` 生成不同产物。

### 命名 lint

生成前会做命名校验：

| 规则 | 示例 |
|---|---|
| `FRONTEND` service 必须以 `Handler` 结尾 | `LobbyHandler` |
| `BACKEND` service 必须以 `Remote` 结尾 | `LobbyRemote` |
| `_Req` 必须配 `_Rsp` | `RPC_Login_Req` -> `RPC_Login_Rsp` |
| `_Ntf` 必须 `returns google.protobuf.Empty` | `RPC_PlayerDisconnect_Ntf` |
| `FRONTEND` input 不允许 `_Rsp` | 防止响应消息被当作上行请求 |
| `cmd_id=0` 不允许用于 message option | 0 保留为“无响应” |

### Handler 生成

`FRONTEND` 生成到：

```text
protocol/gen/handler/<name>_handler.go
```

内容：

- `type XxxHandler interface`。
- `RegisterXxxHandler(d *dispatcher.Dispatcher, srv XxxHandler)`。
- 每个 method 注册到 `dispatcher.Dispatcher`。
- input body 自动 unmarshal。
- `_Req/_Rsp` 方法生成 `corerpc.Reply[*Rsp]` 参数。
- `_Ntf` 方法不带 reply。

### Remote 生成

`BACKEND` 生成到：

```text
protocol/gen/remote/<name>_remote.go
```

内容：

- `type XxxRemote interface`。
- `RegisterXxxRemote(d *dispatcher.Dispatcher, srv XxxRemote)`。

Remote 注册函数负责把 route、unmarshal、Reply 和业务方法绑定到 `dispatcher.Dispatcher`。

### RPC Stub 生成

所有 `BACKEND` service 汇总生成：

```text
protocol/gen/rpc.go
```

内容：

- 包名 `rpc`。
- `GateStub`、`LobbyStub` 等 typed stub。
- 包级变量 `Gate`、`Lobby`。
- `Init(core *corerpc.Core)` 注入 RPC core。
- `At`、`ByHash`、`Broadcast`、`Timeout` 链式 target API。
- `typedCall` / `typedSend` 统一 marshal、调用 core、unmarshal、错误码转换。
- `Join2/Join3/Join4/Each/Seq` 转发到 `internal/core/rpc` 编排 helper。

route 字符串使用 Remote 名称：

```text
GateRemote/SendToClient
LobbyRemote/Login
```

## gen_proto.sh

当前协议生成总入口是 `scripts/gen_proto.sh`：

```bash
# 1. 生成 .pb.go
protoc --proto_path=. --proto_path=/usr/local/include \
    --go_out=. --go_opt=paths=source_relative \
    protocol/common/node.proto \
    protocol/common/options.proto \
    protocol/common/errcode.proto \
    protocol/ra/ra.proto \
    protocol/cs/lobby_cs.proto \
    protocol/ss/lobby_ss.proto \
    protocol/handler/lobby_handler.proto \
    protocol/remote/lobby_remote.proto \
    protocol/remote/gate_remote.proto

# 2. 生成 handler 和 remote stubs
protoc --proto_path=. --proto_path=/usr/local/include \
    --svcstub_out=protocol/gen --svcstub_opt=paths=source_relative \
    protocol/handler/lobby_handler.proto \
    protocol/remote/lobby_remote.proto \
    protocol/remote/gate_remote.proto

# 3. 生成路由表
go run tools/gen_routes/main.go --proto protocol/handler --out protocol/gen/routes.go

# 4. 格式化 generated Go files
gofmt -w protocol/gen/
```

## configgen

`tools/configgen` 从 `config/schema/*.proto` 生成配置代码到 `config/gen`。

输入：

```text
config/schema/*.proto
```

输出：

```text
config/gen/config.go
config/gen/validate.go
config/gen/loader.go
config/gen/entry.go
config/gen/reload.go
```

支持的 schema option 来自 `config/schema/options.proto`，主要用于：

- 标记 root message。
- 标记服务名或 common config。
- 字段 required 校验。
- 字段 reload 行为。
- 字段 env 注入。
- enum values 校验。

限制：

- 同一字段不能同时标记 reload 和 env。
- 生成产物是运行时配置入口，不作为事实源。
- 修改配置结构先改 `config/schema`，再运行 configgen。

## servergen

`tools/servergen` 负责生成服务壳，不生成业务逻辑。

典型用法参数：

```text
--name <svc>              服务名，例如 gamesvr
--pkg <pkg>               internal/server 下包名，默认由 name 去掉 svr 后缀
--kind standard|sidecar   标准服务或 sidecar 服务
--register-env a,b        把服务名追加到 config/values/<env>.yaml 的 svr_list
--dry-run                 只打印计划，不写文件
--force                   覆盖生成器负责的壳文件
```

生成范围：

```text
cmd/<svc>/main.go
cmd/<svc>/CLAUDE.md
internal/server/<pkg>/builder.go
internal/server/<pkg>/config.go
internal/server/<pkg>/options.go
internal/server/<pkg>/CLAUDE.md
config/schema/<pkg>.proto
config/<pkg>.yaml
```

覆盖策略：默认不覆盖已有目标；只有显式 `--force` 才会覆盖生成器负责的文件。

## 构建与配置烘焙工具

当前仓库保留：

- `scripts/config.py`：读取环境值并组织配置烘焙。
- `scripts/config_bake.py`：把模板和环境值渲染成最终配置。
- `scripts/build.py`：按环境编译并铺二进制。

构建与配置烘焙链路以当前 `scripts/` 目录为准；修改时应同步本文件、`scripts/CLAUDE.md` 和 `Makefile`。
