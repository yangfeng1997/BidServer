# 配置系统

本文档记录 BidServer 当前配置系统。配置链路分为四层：schema 事实源、生成代码、环境烘焙、运行时 entry。

## 总览

```text
config/schema/*.proto       配置结构事实源
        │
        ▼
tools/configgen             生成 Go 配置结构、loader、validate、reload、entry
        │
        ▼
config/gen/*.go             生成产物，运行时加载入口
        │
        ▼
scripts/config.py           按环境 values 烘焙运行目录
        │
        ▼
run/<svc>/conf/*.yaml       进程启动时读取的最终配置
        │
        ▼
internal/server/<svc>/      服务 builder 创建 ConfigEntry 并注册 reload hook
```

配置事实源是 `config/schema/`，不是 `config/gen/`，也不是运行目录里的烘焙产物。

## 目录职责

| 路径 | 职责 |
|---|---|
| `config/schema/` | 配置 schema 事实源，使用 proto3 和自定义 option 描述 root、server、required、reload、env、enum_values |
| `config/gen/` | `tools/configgen` 生成的 Go 代码，包含结构体、加载、校验、热更约束和 entry 构造函数 |
| `config/*.yaml` | 配置模板，供 `scripts/config.py` 烘焙 |
| `config/values/*.yaml` | 环境值文件，`svr_list` 决定该环境包含哪些服务 |
| `config/secrets/` | 示例密钥输入或说明，不放真实密钥 |
| `internal/core/config/` | 通用运行时 loader 和 `ConfigEntry` |
| `internal/server/<svc>/config.go` | 服务私有配置 entry 持有、访问函数、reload hook 聚合 |
| `scripts/config.py` | 按环境生成 `run/` 运行配置目录和启动脚本 |
| `scripts/config_bake.py` | 单文件模板烘焙工具 |

## Schema Option

配置 option 定义在 `config/schema/options.proto`：

```protobuf
extend google.protobuf.MessageOptions {
  bool root = 50001;
  bool common = 50002;
  string server = 50003;
}

extend google.protobuf.FieldOptions {
  bool reload = 50005;
  bool required = 50006;
  bool env = 50007;
  string enum_values = 50008;
}
```

Message option：

| option | 作用 |
|---|---|
| `(config.root) = true` | 标记可生成加载入口的根配置 message |
| `(config.common) = true` | 标记公共配置，例如 `CommonConfig` |
| `(config.server) = "gatesvr"` | 标记服务私有配置归属 |

Field option：

| option | 作用 |
|---|---|
| `(config.required) = true` | 生成 `Validate` 必填校验；字符串不可空、数值不可为 0、repeated 不可为空 |
| `(config.reload) = true` | 允许该字段热更 |
| `(config.env) = true` | 标记该字段来自环境注入语义；不能和 `reload` 同时使用 |
| `(config.enum_values) = "a,b"` | 限制 string 字段取值集合 |

约束：同一字段不能同时设置 `(config.reload)` 和 `(config.env)`。`tools/configgen` 会在解析 schema 时拒绝这种组合。

## 当前 Root Config

当前 root 配置：

| 配置 | schema | 模板 | 服务 |
|---|---|---|---|
| `CommonConfig` | `config/schema/common.proto` | `config/common.yaml` | 所有服务共享 |
| `GateConfig` | `config/schema/gate.proto` | `config/gate.yaml` | `gatesvr` |
| `LobbyConfig` | `config/schema/lobby.proto` | `config/lobby.yaml` | `lobbysvr` |
| `RouteragentConfig` | `config/schema/routeragent.proto` | `config/routeragent.yaml` | `routeragent` |

公共结构放在 `config/schema/types.proto`，例如 `LogConfig` 和 `LoggerGroupConfig`。

## configgen 生成产物

`tools/configgen` 扫描 `config/schema/*.proto`，跳过 `options.proto`，生成：

```text
config/gen/config.go
config/gen/validate.go
config/gen/loader.go
config/gen/entry.go
config/gen/reload.go
```

生成内容：

| 文件 | 内容 |
|---|---|
| `config.go` | Go struct 定义和 yaml tag |
| `validate.go` | required、enum_values 和嵌套 message 校验 |
| `loader.go` | `LoadXxx(path)`，调用 `internal/core/config.LoadYAML` 后执行 `Validate` |
| `entry.go` | `NewXxxConfigEntry(path)`，创建 `ConfigEntry` |
| `reload.go` | `CheckXxxReload(candidate, current)`，拒绝未标记 reload 的字段变化 |

生成命令：

```bash
go run ./tools/configgen
```

`Makefile` 的 `gen-config` 目标会执行该命令。

## 模板烘焙

`config/*.yaml` 是模板文件，可以使用 `${key}` 占位符。

环境值来自 `config/values/<env>.yaml`：

```yaml
svr_list:
  - gatesvr
  - lobbysvr
  - routeragent

gate_listen_tcp: "0.0.0.0:7001"
log_dir: "../log"
REDIS_PWD: "dev-password"
```

`scripts/config_bake.py` 的替换规则：

1. 先从 values 文件展开后的 key-value 表查找。
2. 找不到时从进程环境变量查找。
3. 两处都找不到则报错退出。
4. `svr_list` 不进入占位符表，只作为服务列表入口。

嵌套 values 会展平成点分 key；list 会拼成逗号字符串。

示例：

```bash
python scripts/config_bake.py \
  --values config/values/dev.yaml \
  --in config/gate.yaml \
  --out run/gatesvr/conf/gate.yaml \
  --dry-run
```

## 运行目录生成

`scripts/config.py` 负责按环境生成运行目录：

```bash
python scripts/config.py --env dev --out run
```

流程：

1. 读取 `config/values/<env>.yaml`。
2. 从 `svr_list` 得到服务列表。
3. 校验每个服务存在 `cmd/<svc>/` 和对应配置模板。
4. 烘焙 `config/common.yaml` 到 `run/common/conf/common.yaml`。
5. 烘焙每个服务配置到 `run/<svc>/conf/<base>.yaml`。
6. 创建 `run/<svc>/bin`、`run/<svc>/conf`、`run/<svc>/log`。
7. 写入每个服务的 `start.sh` / `stop.sh`。
8. 写入 `run/startall.sh`、`run/stopall.sh` 和 `run/ENV`。

`Makefile` 的 `config` 目标会先执行 `gen-config`，再执行 `scripts/config.py`：

```bash
make config ENV=dev
```

## 运行时加载

服务启动时由 `cmd/<svc>/main.go` 解析配置路径参数，例如：

```text
--common-config ../../common/conf/common.yaml
--gate-config ../conf/gate.yaml
```

服务 builder 负责创建配置 entry：

```go
commonConfig := configgen.NewCommonConfigEntry(opts.CommonConfigPath)
gateConfig := configgen.NewGateConfigEntry(opts.GateConfigPath)
SetCommonConfigEntry(commonConfig)
SetGateConfigEntry(gateConfig)
```

`ConfigEntry[T]` 位于 `internal/core/config/entry.go`：

```go
type ConfigEntry[T any] struct {
    path      string
    load      func(string) (*T, error)
    check     ReloadCheck[T]
    reloadMtx sync.Mutex
    data      atomic.Pointer[T]
}
```

语义：

- `NewConfigEntry` 会立即 `Reload()` 一次，完成启动加载。
- `Get()` 返回当前配置指针。
- `Reload()` 先加载 candidate，再用 generated `CheckXxxReload` 和 current 比较。
- 校验通过后用 atomic pointer 替换当前配置。
- reload 串行化由 `reloadMtx` 保证。

## YAML 加载和运行时环境变量

`internal/core/config.LoadYAML` 会在 unmarshal 前展开 `${NAME}`：

```go
data, err = expandEnv(data)
```

这里查的是进程环境变量。运行时仍存在未替换的 `${NAME}` 会导致加载失败：

```text
env vars not injected: [NAME]
```

通常 `${...}` 应在 `scripts/config.py` 阶段被 values 文件烘焙掉。运行时环境变量展开主要用于保留给部署系统注入的敏感值或机器局部值。

YAML 解码使用 `yaml.Decoder.KnownFields(true)`，配置文件里出现 schema 不认识的字段会报错。

## 热更流程

服务 builder 会把服务自己的 `ReloadConfig` 注册到 `BaseBuilder`：

```go
baseBuilder.AddReloadHook(ReloadConfig)
```

App 收到 reload signal 后执行所有 reload hooks。

每个服务的 `internal/server/<svc>/config.go` 聚合 common config 和服务私有 config：

```go
oldCommon := CommonConfig()
oldGate := GateConfig()

commonConfigEntry.Reload()
gateConfigEntry.Reload()

change := ConfigChange{OldCommon: oldCommon, NewCommon: CommonConfig(), ...}
for _, hook := range configChangeHooks { hook(change) }
```

热更规则由 schema 的 `(config.reload)` 决定：

- 标记 reload 的字段允许变化。
- 未标记 reload 的字段变化会被 `CheckXxxReload` 拒绝。
- 当前 `heartbeat_sec` 标记了 reload。
- logger group、监听地址、容量上限、公共 etcd/redis 配置默认不可热更。

`CommonConfig` 也会参与 reload 检查；当前公共配置字段没有标记 reload，因此公共配置变化会被拒绝。

## 服务侧访问

服务包提供包级访问函数，例如：

```go
gate.CommonConfig()
gate.GateConfig()
lobby.LobbyConfig()
routeragent.RouteragentConfig()
```

这些函数返回当前 `ConfigEntry.Get()` 的结果。配置 entry 在服务 builder 中设置，因此业务代码应在 builder 完成后、模块初始化阶段之后读取。

配置 change hook：

```go
AddConfigChangeHook(func(change ConfigChange) error {
    // compare change.OldGate / change.NewGate
    return nil
})
```

hook 用于服务内组件响应热更结果。hook 返回错误时，`ReloadConfig` 返回错误；当前已经替换成功的 entry 不做自动回滚。

## 构建链路中的位置

常用命令：

```bash
make gen-config
make config ENV=dev
make build
make test
```

`make all` 等价于：

```text
config -> build -> test
```

其中 `config` 会先生成配置代码，再烘焙运行目录。

## 修改规则

新增或修改配置字段时按以下顺序处理：

1. 修改 `config/schema/*.proto`。
2. 必要时修改 `config/*.yaml` 模板。
3. 必要时修改 `config/values/*.yaml` 环境值。
4. 执行 `go run ./tools/configgen` 或 `make gen-config`。
5. 执行 `python scripts/config.py --env dev --dry-run` 检查烘焙。
6. 执行 `go test ./...`。
7. 如果影响目录、入口、schema、热更语义或生成链路，同步更新 `CLAUDE.md` 和本文档。

生成产物 `config/gen/*.go` 不手改；要改生成行为，改 `tools/configgen`。
