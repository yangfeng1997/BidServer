# 协作规则讨论记录

本文件记录与 Claude Code 协作过程中确立的几套规则及其讨论背景。CLAUDE.md 里是精简执行版，本文件保留完整理由与边界讨论，供回溯。注意：按 CLAUDE.md 规则，`docs/TODO/` 默认不被自动读取，本文件仅作讨论存档。

---

## 1. 文档维护约定（keep CLAUDE.md in sync）

**规则**：CLAUDE.md 不是一次生成即固化。代码结构变化时主动同步更新，使其始终反映当前代码现状。

**需要同步的变化**：
- 新增/重命名/删除服务入口（`cmd/`、`internal/server/`）
- 配置 schema（`config/schema/*.proto`）与生成代码链路
- 核心生命周期/进程/日志行为（`internal/core/`、`pkg/logger/`）
- 构建与配置流程（`Makefile`、`tools/*.py`）
- 命令入口、flags、信号语义、热更语义

**无需同步**：函数内部实现、注释、纯重构等不改变架构表面的细节。

**生成产物**：`config/gen/*.go` 变动回溯到 `config/schema/*.proto` 再决定是否更新文档。

**Why**：CLAUDE.md 给后续会话当指引，过时会误导。用户希望它始终反映现状，而非 `/init` 一次固化。

## 2. 默认跳过的目录

**规则**：以下目录默认不读取、不 grep、不遍历，除非用户显式指明。

- `other_projects/`：独立参考项目（`BidKing`、`GameServer`、`ProjectBid`、`cherry`、`pitaya` 等）。仅当用户说"查看参考项目 X"时才进入 `other_projects/X/`。
- `docs/TODO/`：临时方案/讨论文档，非正式参考。仅当用户显式要求时才打开。

**Why**：避免把参考项目代码当当前仓库代码处理；避免把临时讨论当既成事实引用。

## 3. 回答准则：基于行业实际做法

**规则**：回答架构/设计类问题必须基于行业实际做法或实际工业化方向，禁止按理论/教科书方向生搬硬套。本仓库是生产级游戏服务器框架，要可落地的工程判断，不要学术讨论。

**执行要点**：

- **点名具体做法而非抽象优劣**：说"行业用 k8s 多进程 + etcd 服务发现做同类型水平扩展""用 supervisord/systemd 管多进程""Erlang/OTP 的轻量 process 是 VM 级而非 OS 级"，而不是"理论上多进程隔离更好"。给能对应到真实产品/部署模式的名词。
- **分层标注普遍度**：明确区分「行业主流」「小众/特定场景」「理论可行但没人用」，三者不可混为一谈。禁止把罕见路径说成主流，禁止把理论解说成既成事实。
- **不确定就说不确定**：拿不准行业主流时直说"这块我不确定，建议查 X"，不拿训练记忆当事实。点名框架内部实现细节时尤其谨慎——没读源码就不臆测。
- **参考项目不臆测**：`other_projects/` 下的框架（如 `pitaya`）按规则默认不读，回答不引用其内部做法；需要对照时先请用户授权"查看参考项目 X"再读。
- **用实际工程的坑评估方案**：故障隔离、运维心智模型、第三方库进程级全局状态、滚动更新、oncall 调试、资源命名空间冲突等真实代价，而非抽象"优点/缺点"两点式。
- **落到本仓库具体符号**：评估建议落到 `internal/core/app/app.go` 的 `BaseApp`、`internal/server/gate/config.go` 的包级变量、`config/schema/*.proto` 这类具体文件/符号，不泛泛而谈。
- **推荐默认方向**：行业有事实标准时（如游戏服务器同类型扩展走多进程、配置走 codegen、日志走结构化文件），直接给推荐，不把已废弃/罕见路径当备选并列制造决策噪声。

**Why**：用户做生产级框架，需要可落地判断。理论上"更优"但行业没人用的解法，对他就是噪声甚至误导。

## 4. 改动准则：发现结构性冲突先问，别盖补丁

**规则**：发现已有代码/方案与新方案存在**结构性/地基性冲突**时，禁止强行在旧基础上叠补丁（adapter、条件分支、`map` 硬塞状态等）让它凑合工作。必须停下，向用户准确提出冲突与选项，由用户拍板。

### 触发阈值（只在地基冲突时触发）

- **触发**：旧抽象装不下新设计——接口假设与新需求矛盾、生命周期模型不匹配、核心数据结构形状冲突、并发模型冲突等。
- **不触发**：命名、参数顺序、小结构等**表面不匹配**——直接适配，事后说明即可，不打断用户。
- 阈值拿不准时，倾向问一句而非闷头改。

### 准确提出

指明冲突的具体符号（文件/接口/函数/设计假设），说清冲突点是什么，不要笼统说"旧设计不太合适"。

### 给带 tradeoff 的选项（至少考虑四种，不要只给"删 vs 改"二元）

1. **整体替换**：删旧用新——适用于旧的有根本错误、且未被大量调用/未上线。
2. **废弃→迁移→再删**：旧还在用/已上线，先标 deprecated，迁移调用方，再删（big-bang 删有测试/调用方/线上行为代价）。
3. **Strangler 留壳**：旧接口保留转调新实现，新逻辑走独立路径——业内重构大型系统的主流做法。
4. **适配旧承载新并注明理由**：若旧设计确实可承载、只是不优雅，可适配，但代码注释说明为什么没推倒。

### 中立表述

旧代码可能体现用户有意的决策，不预设"旧的是坏的要删"。提问是中立的"这里冲突，选项如下"，不是"旧设计不好，删掉？"。

### 覆盖范围

与系统已有原则"覆盖/删除前先看目标、发现矛盾要 surface"一脉相承，但范围从"删文件"扩展到"架构适配决策"——任何会塑造架构走向的适配方式选择，都要先问。

### 本仓库示例

假设新方案要给 gate 加每连接生命周期。当前 `Module` 接口（`internal/core/app/module.go`）是**应用级单例**：`Init`/`Shutdown` 每个 App 跑一次，没有 per-conn 钩子。

- **错误做法（违反本规则）**：在 `GateModule` 里塞 `map[connID]*session`，加一堆 `if session != nil` 条件，把连接状态硬塞进单例 module——补丁越摞越厚。
- **本规则要求**：停下来指出——"`Module` 接口假设应用级单例生命周期，新方案的每连接生命周期装不下。选项：(a) 新增 `Session` 抽象层；(b) 扩展 `Module` 接口加 per-conn 钩子（影响所有现有 module）；(c) Strangler：旧 module 留壳，新连接逻辑走独立路径。你定哪个？"

**Why**：在错地基上盖补丁会 accrete 成系统性债务；旧设计可能是有意决策，强改会破坏意图。停在冲突点让用户拍板，是控制架构走向不漂移的关键。
