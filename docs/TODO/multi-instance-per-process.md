# 单进程同类型多实例的可行性与坑

讨论：单进程内跑两份同类型 server 实例（如两个 gatesvr 在一个进程里），行业是否这么做？后期坑是否兜得住？

---

## 行业真这么做吗？

**同类型多实例单进程：几乎没有。** 解决不了任何真实问题：

- **要扩展？** 行业是**多进程**扩展，不是单进程塞多实例。gate 不够用就起 `gatesvr1`/`gatesvr2`/`gatesvr3` 三个进程，前面挂 LB 或靠 etcd 服务发现 + 一致性哈希。这是 k8s/systemd 标准玩法。
- **要隔离？** 多实例单进程反而**丢掉**隔离——正是要分开的初衷。
- **要省内存？** 游戏服务器业务量大，进程基线开销占比很小，不值得为省这点内存冒耦合风险。

行业里"多实例单进程"的常见形态是**一个 App 内部托管多个房间/玩家**（棋牌、MOBA、MMO zone），那是"一个 server 实例 + 多个业务对象"，不是"多个 server 实例"。

**不同类型同进程**（gate+lobby 塞一个进程）也基本只用于本地联调的 "all-in-one" dev binary，生产很少，同样牺牲隔离。

---

## 后期的坑——为什么会兜不住、乱

多实例单进程的坑**累积叠加**，不是单点问题：

### 1. 资源命名空间冲突

- 端口：两个 gate 实例得用不同 listen 端口——那"同类型"就没意义了，只是造了两个配置不同的实例。
- 日志文件：`gatesvr.log` basename 撞车，得给每个实例加 id 前缀，所有日志路径都要参数化。
- PID 文件、pprof 端口（6060）、etcd 注册的 service name、prometheus 指标——全要分区，漏一处就串。

### 2. 第三方库的进程级全局

就算把自己代码的包级变量全去掉，Go 标准库和第三方库仍满是进程级单例：`http.DefaultServeMux`、`prometheus.DefaultGatherer`、grpc 默认 server、`runtime/pprof`。两个实例默认共享，踩坑无声无息。

### 3. 无故障隔离（最致命）

一个实例 panic/OOM/死循环 → 整个进程跪，另一个陪葬。把"分开部署为了不一起死"的初衷彻底反向了。Go 的 GC pause 也是全进程的，A 触发大 GC，B 一起卡。

### 4. 无法独立生命周期

不能只重启 A 不重启 B；滚动更新变成全有或全无；不能 drain A 同时让 B 继续服务；`signal.Notify` 是进程级，SIGTERM 一发两个实例都收到，想单独停一个得自己造 admin API——更多移动部件。

### 5. 调试/运维模型崩塌

- 一个 pprof、一份 goroutine dump 混两个实例，CPU 飙高不知道是谁。
- 指标端点混采，日志交错（除非每个字段都打 instance_id，漏一个就分不清）。
- oncall 看到 `gatesvr` 进程吃 2 倍内存/CPU，分不清是"两个实例"还是"一个实例忙"，心智模型断了。**"server 实例 = 进程"这个简单映射一旦破坏，运维复杂度永久上升。**

### 6. 隐性耦合会蔓延

人在同进程里，迟早有人"顺手优化"：让 A 和 B 直接共享内存/chan 通信，绕过网络。一旦这么干，A、B 就耦合死了，以后想拆回多进程要重构。这是"乱"的来源——**不是一次性设计错，是会持续 accrete 的债务。**

### 7. 扩展不对称

有的进程 1 实例、有的 2 实例，部署脚本要分支处理，自动化运维的统一性没了。

## 什么时候才真值得

极少数场景：

- 无状态微型服务、进程开销占比极高（游戏服务器基本不在此列）。
- 硬实时/嵌入式，OS 进程模型不可用。
- 纯 dev/test 联调二合一。

游戏服务器生产：**不属于这些场景**。

---

## 给这套框架的明确建议

- **别为"同进程多实例"做架构改造**。去掉包级全局变量的目标是**可测试 + 依赖显式**，不是为生产多实例铺路。别把"能多实例"当目标去追——那是把 dev 便利性当生产架构，本末倒置。
- **同类型水平扩展 = 多进程**。`gatesvr1.pid`/`gatesvr2.pid` 各自独立进程，配 etcd 注册 + 前端 LB。`startall.sh` 现在的多进程方向就是对的，延续即可。
- **多租户需求 = 一个 App 内部 rooms/players**，不是多 App。
- 真要本地联调把 gate+lobby 塞一个进程，那是 dev 专用二进制，生产不这么做，代码里要明确标注 "dev only"。

## 一句话

同类型多实例单进程，行业基本不做；做的那些后期都被资源冲突、无隔离、共享全局、耦合蔓延的坑拖累，越来越乱。当前框架的正确方向是**多进程**，不是改造成多实例。
