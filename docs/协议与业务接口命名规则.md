# 协议与业务接口命名规则

本文档约定客户端协议与服务间 RPC 在业务接口层的命名，避免后续生成工具、目录结构和业务实现出现混用。

## 命名结论

- 客户端协议入口业务接口称为 `Handler`。
- 客户端协议入口分发服务称为 `HandlerService`。
- 服务间 RPC 业务接口称为 `Remote` 或 `RemoteService`。
- 当前仓库协议目录中，`protocol/handler` 对应客户端入口，`protocol/remote` 对应服务间 RPC。

## 客户端协议：Handler

客户端发来的 request / notify 进入网关后，会被解析为客户端入口消息，再按路由交给业务处理。

这一层业务接口统一叫 `Handler`，例如：

```text
LobbyHandler
GateHandler
```

对应目录与生成物：

```text
protocol/handler/
protocol/gen/handler/
internal/core/dispatcher/
```

适用场景：

- 客户端请求业务服，例如领取奖励、同步位置。
- 客户端通知业务服，例如心跳、状态同步。
- 需要按 `cmd_id` 建立客户端入口路由表。

## 服务间 RPC：Remote / RemoteService

服务之间的调用属于后端 RPC，不直接表示客户端协议入口。

这一层业务接口统一称为 `Remote`。protobuf 中仍使用 `service` 语法声明 RPC，但目录、生成代码和业务接口都使用 `Remote` 命名。

当前仓库使用：

```text
protocol/remote/
protocol/gen/remote/
protocol/gen/rpc.go
internal/core/rpc/
```

适用场景：

- gatesvr 调 lobbysvr。
- lobbysvr 调其他后端服务。
- routeragent 转发服务间请求、响应或通知。

## 与 Pitaya 的对应关系

Pitaya 中的命名可以作为参考：

```text
客户端协议业务接口：Handler / HandlerService
服务间 RPC 业务接口：Remote / RemoteService
```

因此当前仓库中不要把客户端入口叫 remote，也不要把服务间 RPC handler 和客户端 handler 混在同一层。

## 目录约定

建议保持以下边界：

```text
protocol/handler/       客户端入口 handler proto
protocol/gen/handler/   客户端入口 handler 生成代码
protocol/remote/       服务间 RPC remote proto
protocol/gen/remote/   服务间 RPC remote 生成代码
protocol/gen/rpc.go     服务间 RPC typed stub
internal/core/dispatcher 客户端入口消息分发
internal/core/rpc        服务间 RPC core
```

## 生成工具约定

- `tools/gen_routes` 只面向客户端入口 handler proto，生成客户端入口路由表。
- `tools/protoc-gen-svcstub` 可以生成 handler stub、remote stub 和 typed RPC stub，但生成结果要保持 handler 与 remote 分层。
- 新增协议时，先判断它是客户端入口还是服务间 RPC，再选择放入 `protocol/handler` 或 `protocol/remote`。
