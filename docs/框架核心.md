# 框架核心

本文档记录 BidServer 框架核心实现。当前框架以 `BaseBuilder` 组装进程，以 `BaseApp` 管理模块生命周期、信号、reload、pprof、主循环投递和协程等待。

## 核心目录

当前框架核心位于 `internal/core`：

| 目录 | 职责 |
|---|---|
| `app/` | App 生命周期、Module 管理、Post 主循环投递、Ready 等待、signal、reload hook、shutdown hook |
| `options/` | 通用启动参数结构 |
| `process/` | daemon、pidfile、signal 分类 |
| `logger/` | 框架日志组构建和全局 logger |
| `config/` | 框架配置辅助能力 |
| `nodeid/` | NodeID 编解码 |
| `acceptor/`、`conn/`、`codec/` | 客户端网络接入基础设施 |
| `dispatcher/`、`session/` | 客户端消息分发和会话管理 |
| `rpc/` | 服务间 RPC core、Ctx、Target、编排 helper |

可复用公共库放在 `pkg/`，例如 `pkg/taskqueue`、`pkg/logger`、`pkg/timewheel`、`pkg/event`、`pkg/serialize`。

## App 与 Builder

当前进程入口通常是：

```text
cmd/<svc>/main.go
  -> 解析 flags
  -> 处理 daemon / pidfile
  -> internal/server/<svc>.New<Svc>Builder(opts)
  -> builder.Build()
  -> app.Startup()
```

`internal/core/app.Builder` 接口：

```go
type Builder interface {
    Build() (App, error)
    AddModule(module Module)
    AddShutdownHook(hook func())
    AddReloadHook(hook func() error)
    SetDaemon(daemon bool)
    SetPprof(enabled bool, addr string)
}
```

各服务 builder 负责进程级装配：

1. 加载 common config 和服务私有 config。
2. 设置当前服务的 config entry。
3. 根据配置创建 `logger.LoggerGroup`。
4. 创建 `app.BaseBuilder`。
5. 设置 daemon、pprof。
6. 注册需要的 Module、shutdown hook、reload hook。

例如 `routeragent` builder 会额外 `AddModule(NewModule())`。`gate`、`lobby` 的网络、业务模块按服务职责在对应 builder 中继续装配。

## Module 生命周期

当前 `internal/core/app.Module`：

```go
type Module interface {
    Name() string
    App() App
    Set(app App)
    Init() error
    AfterInit() error
    BeforeShutdown()
    Shutdown()
}
```

`BaseModule` 提供空实现，并保存 `App` 引用。

启动流程由 `BaseApp.Startup` 串行执行：

```text
1. startPprof()
2. 按注册顺序调用 Set(app)
3. 按注册顺序调用 Init()
4. 按注册顺序调用 AfterInit()
5. 按注册顺序等待实现 ReadyWaiter 的模块 WaitReady(ctx)
6. running = true
7. 进入 runLoop()
8. runLoop 退出后 Shutdown()
9. 逆序 BeforeShutdown()
10. 逆序 Shutdown()
11. close dieNotifyChan
12. 等待 AddRoutine/DoneRoutine 管理的 goroutine
13. 逆序执行 shutdown hooks
```

`BeforeShutdown` 和 `Shutdown` 分工：

| 阶段 | 顺序 | 职责 |
|---|---|---|
| `BeforeShutdown` | 模块注册逆序 | 停止对外接入、关闭监听、通知后台 goroutine 停止 |
| `Shutdown` | 模块注册逆序 | 释放内部资源 |
| shutdown hooks | hook 注册逆序 | 关闭 logger group、pprof server 等非 Module 资源 |

所有 Module 通过同一个 `AddModule` / `RegisterModule` 列表管理，执行顺序就是注册顺序。

## ReadyWaiter

异步初始化模块可以实现：

```go
type ReadyWaiter interface {
    WaitReady(ctx context.Context) error
}
```

`internal/core/app/ready.go` 提供 `Ready`：

```go
ready := app.NewReady()
ready.Done()
ready.Fail(err)
err := ready.WaitReady(ctx)
```

`BaseApp.waitReady` 会给每个 ReadyWaiter 一个 10 秒 timeout。当前 `routeragent.Module` 使用 `app.Ready` 等待 UDS/TCP 初始化结果。

## 主循环与 Post

当前主循环是事件驱动模型：

```go
for {
    select {
    case fn := <-app.postQueue.C():
        fn()
    case <-app.dieChan:
        break
    case sig := <-app.sigChan:
        // terminate / drain -> shutdown
        // reload -> app.Reload()
    }
}
```

`App.Post(fn)` 是跨 goroutine 回到主循环的统一入口：

```go
type Poster interface {
    Post(fn func())
}
```

网络 read goroutine、timer callback、etcd watch callback、RPC 回包等跨 goroutine 事件，都应通过 `Post` 回到主循环执行共享状态修改。

当前 `BaseApp` 没有 tick / Updater 轨道，也没有在主循环内推进 timewheel。需要周期逻辑的服务可以先用 goroutine/timer 回投 `Post`，后续如引入统一 tick，应同步修改本文件和 `internal/core/app/CLAUDE.md`。

## taskqueue

`BaseApp` 内置 `pkg/taskqueue.Queue`：

```go
type Queue struct {
    ch          chan func()
    lastFullLog atomic.Int64
    fullHits    atomic.Uint64
}
```

特性：

- channel-based MPSC：多个生产 goroutine，一个 App 主循环消费者。
- 默认容量 256。
- `Post` 在队列未满时直接写入。
- 队列满时限频打印日志，并阻塞生产者形成背压。
- 不丢任务，避免 RPC 回包、连接事件等关键回调丢失。

## 信号、Reload 与停服

`runLoop` 监听 `process.WatchedSignals()`：

- terminate signal：退出主循环，进入普通停服。
- drain signal：当前也进入停服流程，后续可扩展为先停接入再等待存量。
- reload signal：调用 `app.Reload()`，执行 reload hooks，主循环继续。

`Reload()` 会按注册顺序执行 `reloadHooks`。各服务 builder 当前把服务配置 `ReloadConfig` 注册为 reload hook。

## goroutine 管理

非 Module 管理的后台 goroutine 可以通过 App 记录生命周期：

```go
app.AddRoutine(1)
go func() {
    defer app.DoneRoutine()
    // work
}()
```

`Startup` 退出阶段会 close `dieNotifyChan`，然后等待 WaitGroup。后台 goroutine 应监听 `DieNotifyChan()` 或自己的 stop channel，避免停服卡住。

## 配置与日志边界

当前仓库没有把 config 和 logger 做成 App Module。它们由各服务 builder 在 Build 前完成初始化：

- 配置 entry 由 `config/gen` 生成并由 `internal/server/<svc>/config.go` 持有。
- 服务 builder 先加载配置，再创建 `logger.LoggerGroup`。
- logger group 的 `Shutdown` 注册为 shutdown hook。
- 服务配置的 `ReloadConfig` 注册为 reload hook。

如果后续把 config 或 logger 改为 Module 化，需要同步更新服务 builder、启动流程和文档。

## 依赖获取规则

当前建议：

| 依赖类型 | 获取方式 |
|---|---|
| App / Poster | Module 的 `Set(app)` 后可通过 `App()` 获取；需要跨 goroutine 投递时断言为 `app.Poster` |
| 业务模块依赖 | 在服务 builder 或业务构造函数中显式传入 |
| 运行配置 | 通过当前服务的 config entry 包级函数读取 |
| 日志 | 通过 `internal/core/logger` 或 logger group 初始化后的全局 logger 使用 |
| 兜底查找 | `App.GetModule(name)`，找不到返回 error |

避免在 goroutine 中直接修改共享业务状态。跨 goroutine 结果统一 `Post` 回主循环。
