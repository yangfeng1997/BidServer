# 集群通信：RouterAgent

RouterAgent 是本框架的集群通信 sidecar。业务进程通过本机 UDS 连接 RouterAgent，跨机器通信由 RouterAgent 之间的 TCP 连接完成。

## 物理拓扑

```text
┌─────────────────────────────────────────────┐
│  物理机 A                                    │
│                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ gatesvr  │  │ lobbysvr │  │  其他进程 │  │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  │
│       │UDS           │UDS          │UDS      │
│       └──────────────┴─────────────┘        │
│                      │                      │
│              ┌───────▼────────┐             │
│              │  RouterAgent A │◄──── etcd   │
│              └───────┬────────┘             │
└──────────────────────┼──────────────────────┘
                       │ TCP
```

当前实现目录是 `internal/server/routeragent`。默认 UDS 路径为：

```text
/run/routeragent/ra.sock
```

业务进程连接后先发送 Handshake，携带自己的 NodeID。RouterAgent 用 NodeID 区分本机不同业务连接。

## 并发模型

RouterAgent 作为一个 App Module 运行，复用 `internal/core/app` 生命周期：

- `AfterInit` 启动 UDS server、keepalive 和可选 TCP server。
- UDS/TCP read goroutine 收到帧后通过 `app.Poster` 投递到 App 主循环。
- 路由决策、成员表读写、remote seq 映射和广播 waiter 都在主循环任务中处理。
- `WaitReady` 通过 `app.Ready` 等待 UDS/TCP 初始化结果。

当前 App 主循环是事件驱动模型，没有独立 tick 轨道。RouterAgent 的超时类逻辑应使用 `time.AfterFunc` 或 goroutine 回投主循环，避免在 timer goroutine 里直接修改共享状态。

## UDS 帧格式

当前 `frame.go` 使用统一帧格式：

```text
| length(4B, big-endian) | type(1B) | header_len(2B) | header | body |
```

其中：

- `length` 不包含自身 4 字节。
- `header_len` 最大 `uint16`。
- `RpcRequest`、`RpcResponse`、`RpcNotify` 使用 header 承载 RPC 路由头。
- body 是业务 payload 或控制消息 body。

帧类型：

| 值 | 名称 | 说明 |
|---|---|---|
| `0x01` | `Handshake` | 业务进程连接 RouterAgent 后声明 NodeID |
| `0x02` | `HandshakeAck` | RouterAgent 确认注册结果 |
| `0x03` | `RpcRequest` | 服务间请求 |
| `0x04` | `RpcResponse` | 服务间回包 |
| `0x05` | `RpcNotify` | 单向通知 |
| `0x06` | `Heartbeat` | UDS 链路保活 |
| `0x07` | `BroadcastSent` | 广播投递结果通知 |

## RPC Wire Header

当前 `rpc_wire.go` 的头部字段为：

```go
type RPCWireHeader struct {
    SeqID       uint64
    ServerType  uint32
    RoutingMode uint8
    DeadlineMs  int64
    WaiterID    uint64
    FromNodeID  uint32
    ErrCode     uint32
    RoutingKey  string
    Route       string
}
```

RouterAgent 主要读取 `ServerType`、`RoutingMode`、`RoutingKey`、`SeqID`、`FromNodeID` 和 `Route`。业务 payload 不在 RouterAgent 内解析。

## 路由模式

当前 RouterAgent 的路由模式定义在 `resolver.go`：

| 模式 | 说明 |
|---|---|
| `RoutingModeAny` | 从目标 serverType 的节点列表中选择一个节点 |
| `RoutingModeDirect` | 从 `RoutingKey` 解析完整 NodeID，直达该节点 |
| `RoutingModeHash` | 用 `RoutingKey` 对目标 serverType 节点集合取哈希 |
| `RoutingModeBroadcast` | fanout 到目标 serverType 的全部节点 |

当前哈希选择是对按 NodeID 排序后的列表取 FNV hash 余数，不是 ketama 一致性哈希。后续如要减少节点增删扰动，可以替换 `Resolver.PickHash` 实现。

## 本地短路与跨机转发

路由处理流程：

1. 根据 RPC header 选择目标 NodeID 列表。
2. 通过成员表查询目标 NodeID 对应的 RA 地址。
3. 如果目标 RA 地址是本机连接，则直接投递到本地 UDS 连接。
4. 如果目标在其他机器，则通过 `PeerMgr` 获取或建立 TCP peer 连接。
5. 跨机请求分配 remote seq，并在 `RemoteSeqMap` 里记录原始 UDS 连接和原始 seq。
6. 远端回包返回时再把 remote seq 映射回原始 seq，投递给原始业务进程。

## 服务注册与发现

当前仓库已经有 `etcd_registry.go`，提供：

- `Register(nodeID, raAddr, serverType)`：写入节点信息并用 lease keepalive。
- `Discover(onAdd, onDel)`：全量加载并 watch 节点变化。
- `PeerAddr(nodeID)`：按 NodeID 查询 RA 地址。

RouterAgent 的服务发现由 `EtcdRegistry` 负责，模块装配时应把注册、全量发现和 watch 结果接入 `MemberTable`，并确保所有回调通过 `app.Poster` 回到主循环。

建议 key 语义保持为 NodeID 主键：

```text
/routeragent/nodes/{nodeID}
value: { node_id, server_type, ra_addr, start_at }
```

## Keepalive 语义

`keepalive.go` 只负责 RouterAgent 与本机业务进程之间的 UDS 链路健康。它和客户端心跳不是同一概念：

- RouterAgent keepalive：检测本机业务进程是否仍健康连接 sidecar。
- 客户端 heartbeat：检测玩家客户端连接是否仍存活。

命名和指标上应保持这两个概念分离。
