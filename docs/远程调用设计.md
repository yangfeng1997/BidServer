# RPC 设计

本文档记录当前服务间 RPC 的设计和实现状态。当前仓库已经统一使用 `Remote` 表示服务间 RPC 业务接口，客户端入口仍称为 `Handler`。

## 核心目标

- 跨进程调用异步执行，不阻塞 App 主循环。
- 回包和超时都回投到 App 主循环，业务回调串行执行。
- 业务使用强类型 generated stub，不直接拼 route、marshal 或 unmarshal。
- RouterAgent 只做路由和转发，不解析业务 payload。

## RPC Core

核心实现位于 `internal/core/rpc`：

```go
type Transport interface {
    SendFrame(target Target, header Header, body []byte) error
}

type Core struct {
    transport Transport
    poster    Poster
    timeout   time.Duration
    seq       atomic.Uint64
    mu        sync.Mutex
    pending   map[uint64]*inflight
}
```

`Core.Call` 用于请求响应型 RPC：

1. 分配 seq。
2. 记录 pending 回调。
3. 根据 `Ctx` deadline、默认 timeout 和 `Target.Timeout` 计算超时。
4. 用 `time.AfterFunc` 触发超时。
5. 通过 `Transport.SendFrame` 发出帧。
6. 收到 `OnResponse` 后删除 pending、停止 timer、结束 span，并把业务回调投递到 `Poster`。

`Core.Send` 用于单向通知：

- 不分配 seq。
- 不记录 pending。
- 不挂超时回调。
- 直接通过 transport 发出。

当前 `pending` 使用 mutex 保护，允许回包和 timer 从不同 goroutine 进入；业务回调仍通过 `Poster` 回到主循环执行。

## Ctx

`rpc.Ctx` 承载跨跳传播信息，不承载路由意图：

| 字段 | 作用 |
|---|---|
| `deadline` | 调用剩余时间预算 |
| `span` | 轻量链路跨度 |
| `fromNode` | 来源 NodeID |
| `clientMeta` | 客户端入口附带信息 |
| `stale` | 实体失效保护 |

路由意图只放在 `Target` 或 generated stub 的链式调用对象上，例如 `At`、`ByHash`、`Broadcast`、`Timeout`。

## Target 和路由 API

`Target` 定义在 `internal/core/rpc/types.go`：

```go
type Target struct {
    ServerType uint32
    Mode       RoutingMode
    Key        string
    NodeID     uint32
    Deadline   time.Duration
}
```

链式 API：

```go
rpc.Lobby.Login(ctx, req, cb)
rpc.Lobby.At(nodeID).Login(ctx, req, cb)
rpc.Lobby.ByHash(key).Login(ctx, req, cb)
rpc.Lobby.Broadcast().PlayerDisconnect(ctx, ntf)
rpc.Lobby.Timeout(3 * time.Second).Login(ctx, req, cb)
```

`At/ByHash/Broadcast/Timeout` 返回带新 target 的副本，不修改包级默认 stub。

## 入站分流

业务进程收到 RouterAgent 帧后应按帧类型分流：

| 入站类型 | 含义 | 去向 |
|---|---|---|
| `RpcRequest` / `RpcNotify` | 别人调用我 | `dispatcher.Dispatch`，进入 Handler 或 Remote 注册函数 |
| `RpcResponse` | 我之前调用别人的回包 | `rpc.Core.OnResponse`，按 seq 找 pending |

回包不走 dispatcher。dispatcher 只处理新的入站请求或通知。

## Handler 与 Remote

proto service 分两类：

```protobuf
service LobbyHandler {
  option (protocol.common.kind) = FRONTEND;
  option (protocol.common.server_type) = ST_LOBBYSVR;

  rpc ClaimReward(CS_ClaimReward_Req) returns (SC_ClaimReward_Rsp);
  rpc SyncPos(CS_SyncPos_Ntf) returns (google.protobuf.Empty);
}

service LobbyRemote {
  option (protocol.common.kind) = BACKEND;
  option (protocol.common.server_type) = ST_LOBBYSVR;

  rpc Login(RPC_Login_Req) returns (RPC_Login_Rsp);
  rpc PlayerDisconnect(RPC_PlayerDisconnect_Ntf) returns (google.protobuf.Empty);
}
```

约定：

- `FRONTEND` 生成 `protocol/gen/handler` 下的 Handler 接口和注册函数。
- `BACKEND` 生成 `protocol/gen/remote` 下的 Remote 接口和注册函数。
- 所有 BACKEND 汇总生成 `protocol/gen/rpc.go` 的 typed stub。
- route 字符串使用 `RemoteName/MethodName`，例如 `LobbyRemote/Login`。

## 消息命名规范

| 后缀 | 含义 | 示例 |
|---|---|---|
| `_Req` | 请求，期待回包 | `CS_Login_Req`、`RPC_Login_Req` |
| `_Rsp` | 响应 | `SC_Login_Rsp`、`RPC_Login_Rsp` |
| `_Ntf` | 单向通知 | `CS_SyncPos_Ntf`、`RPC_PlayerDisconnect_Ntf` |

前缀约定：

- `CS_`：Client to Server。
- `SC_`：Server to Client。
- `RPC_`：服务间 RPC。

单向与双向由命名和返回类型共同决定：

```text
input _Ntf && returns google.protobuf.Empty  -> Send
input _Req && returns *_Rsp                  -> Call
其他组合                                      -> 生成器报错
```

空响应也应定义空字段 `_Rsp` message，而不是使用 `google.protobuf.Empty`。`Empty` 表示单向通知。

## Generated Stub

当前 `protocol/gen/rpc.go` 生成：

- `GateStub`、`LobbyStub` 等 typed stub。
- 包级变量 `rpc.Gate`、`rpc.Lobby`。
- `Init(core *corerpc.Core)` 注入 RPC Core。
- `Join2/Join3/Join4/Each/Seq` 编排辅助函数转发。

示例：

```go
rpc.Init(core)
rpc.Lobby.Login(ctx, req, func(rsp *remote.RPC_Login_Rsp, err error) {
    // 回调由 rpc.Core 通过 Poster 投递回主循环
})
rpc.Gate.At(gateNodeID).SendToClient(ctx, ntf)
```

`protocol/gen/remote/*.go` 生成 Remote 接口和注册函数。Remote 入站注册应把 route、unmarshal、Reply 和业务方法绑定到 `dispatcher.Dispatcher`，使服务间请求落到对应 Remote 实现。

## Reply 与编排

`corerpc.Reply[T]` 是一次性回包句柄类型：

```go
type Reply[T any] func(T, error)
```

当前 Handler 生成代码已经把 `_Req/_Rsp` 方法生成为携带 Reply 的接口签名。Remote 接口也使用相同签名规则。

编排辅助位于 `internal/core/rpc/compose.go`：

- `Join2/Join3/Join4`：并行等待固定数量异构调用。
- `Each`：对多个同类目标执行同一调用。
- `Seq`：顺序执行异步步骤。

当前这些 helper 使用 atomic 计数或回调链组织结果。调用方仍应确保最终业务回调回到主循环执行。

## 错误码

错误码定义在 `protocol/common/errcode.proto`，Go 包装在 `internal/core/errcode`。

跨服务 RPC 回包使用 `ErrCode` 表示框架级或业务级错误。生成 stub 中如果 `code != errcode.OK`，会转换为 `errcode.From(code)` 传给业务 callback。
